# atuin-api-client

A client for the atuin sync server HTTP API, generated by `build.rs` with
[progenitor](https://github.com/oxidecomputer/progenitor) from the OpenAPI
specification at `crates/atuin-client/openapi.json`.

That spec is itself generated, from `atuin-server`:

```
atuin-server handlers + atuin-common types   (source of truth)
  -> crates/atuin-client/openapi.json        (utoipa, OpenAPI 3.0.3, committed)
  -> $OUT_DIR/codegen.rs                     (progenitor, at build time)
```

Nothing in this crate is written by hand, and no generated code is committed:
`src/lib.rs` only `include!`s the build output. `build.rs` reruns whenever the
spec changes, so a normal `cargo build` is enough to pick up an API change.

Like `atuin-daemon`, which commits `proto/*.proto` but generates its Rust at
build time, the contract is committed and the code is not.

## Changing the API

After changing a handler, a route, or a wire type in `atuin-server` /
`atuin-common`, regenerate the committed spec from the repository root:

```sh
cargo run -p atuin-server --bin atuin-server -- openapi > crates/atuin-client/openapi.json
```

The client then regenerates on the next build.
`crates/atuin-server/tests/openapi.rs` fails if the committed spec drifts from
the server, or if a route in `router.rs` is missing from the `paths(..)` list in
`atuin-server/src/openapi.rs`.

Note that progenitor parses OpenAPI 3.0.x only, so `utoipa` must stay on 4.x
(5.x emits 3.1.0) unless a downconvert step is added.

## Consumers

`atuin-client/src/api_client.rs` injects a configured `reqwest::Client` via
`Client::new_with_client` to set the `Authorization` header, `User-Agent`,
`Atuin-Version` header, and network timeouts. The base URL may include a path
prefix and must not end with a trailing slash.
